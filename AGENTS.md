# 서보 브리지 개발 지침

## 프로젝트 목적

- 이 저장소는 ESP32-C3/PCA9685 펌웨어와 ROS2 Python 브리지를 함께
  개발하여 실제 서보 장치, 직접 제어 UI 및 로봇 시뮬레이터를 동일한 제어
  경로로 연결하는 것을 목적으로 한다.
- 전체 데이터 흐름은
  `Isaac Sim 또는 MuJoCo -> ROS2 -> Python 브리지 -> USB Serial -> ESP32-C3 -> PCA9685`
  로 구성한다. 직접 제어 UI도 ROS2 인터페이스를 통해 같은 경로를 사용한다.
- ROS2 기준 배포판은 Jazzy이다. Windows 10/11과 Ubuntu 24.04를 모두 정식
  지원하며 현재 주 개발 및 실제 장치 검증 환경은 Windows이다.
- ESP32 시리얼 프로토콜을 실제 하드웨어 제어의 단일 기준으로 유지한다.
  ROS2 노드와 UI가 PCA9685 또는 서보를 우회하여 직접 제어하지 않는다.
- 각 채널은 로봇 관절뿐 아니라 그리퍼나 보조 액추에이터일 수 있다. 특히
  7번과 8번 채널을 관절로 가정하지 않으며 역할은 설정 파일에서 정의한다.
- 실제 위치 센서가 추가되기 전까지 ESP32의 `STATE`와 ROS2
  `JointState`는 측정값이 아니라 펌웨어가 계산한 open-loop 상태임을
  명확히 표시한다.

## PCA9685 출력 요구사항

- 기존 시리얼 명령 프로토콜과 채널 번호 체계를 유지한다.
- `PWM_INITIALIZATION_DEBUG`가 활성화되면 부팅 초기화와 각 시리얼 `#RESET`
  동작에서 실제 PCA9685 `setPWM()` 쓰기를 `PWMDBG` 형식으로 출력한다.
  `phase=BOOT` 또는 `phase=RESET`을 포함하며 일반 제어 명령은 출력하지
  않는다. 각 구간이 끝나면 누적 쓰기 횟수를 출력한다.
- cosine 모션 보간, 채널별 독립 수행시간, 지연 실행 및 마지막 데드밴드
  보정 단계를 유지한다.
- PCA9685 PWM 설정 주파수는 50 Hz를 유지한다.
- 25 MHz 발진기와 50 Hz 설정의 실제 분주 주파수를 기준으로 마이크로초
  단위 펄스 폭을 PCA9685 카운터 값으로 변환한다.
- 데드밴드와 마지막 보정은 실제 PCA9685 카운터 단위로 판단한다.
- 채널별 가동 범위, 방향, 오프셋, 초기 위치 및 각도 제한 설정을 유지한다.
- 부팅과 `#RESET`은 모터 8번부터 1번까지 역순으로 초기화하며 각 모터의
  이동 시작 간격은 1000ms이다. 각 이동은 기존 cosine 보간과 속도 제한을
  유지한다.
- 부팅 시 PCA9685 출력을 먼저 끄고 역순으로 초기 PWM을 인가한다. 마지막
  1번 모터의 초기 PWM 인가가 시작된 뒤 `READY`를 출력하며, 그전에는
  시리얼 명령을 처리하지 않는다.
- `#RESET`은 진행 중인 모션과 모든 지연 명령을 현재 계산 위치에서 취소한
  뒤 순차 리셋을 예약하고 `OK`를 즉시 반환한다.
- `#RESET`으로 예약된 각 모터가 실제로 시작될 때
  `RESET motor8 angle:90.00` 형식의 메시지를 8번부터 1번까지 출력한다.
  부팅 초기화와 일반 지연 명령에서는 이 메시지를 출력하지 않는다.
- 일반 루프의 모션 갱신 시각은 시리얼 명령 처리가 끝난 뒤 읽는다. 새 명령의
  모션 시작 시각보다 오래된 `millis()` 값으로 갱신하여 unsigned 시간 계산이
  언더플로되지 않도록 유지한다.
- PCA9685 한 개로 최대 16개 채널을 지원한다. 채널을 늘릴 때는
  `SERVO_COUNT`, `SERVO_CHANNELS` 및 모든 채널별 설정 배열을 함께 늘린다.
- 각 서보 채널은 관절축뿐 아니라 그리퍼 같은 보조 액추에이터에 대응할 수
  있다. 특히 7번과 8번을 관절축으로 단정하지 않는다.
- PCA9685 채널 0은 사용하지 않는다. 모터 번호와 PCA9685 채널 번호는
  항상 동일하게 유지하며 단일 보드에서는 모터 1~15를 사용할 수 있다.
- 오류 응답은 `ERR,<영문 코드>,<한국어 설명>,INPUT=<입력>` 형식을
  유지하고 시리얼 클라이언트는 UTF-8로 디코딩한다.
- 일반 각도 및 Raw PWM 명령이 정상 등록되면 `OK` 다음 줄에 선택된 모터를
  번호순으로 출력한다. 각도 항목은 입력 각도와 함께 PCA9685 카운터로
  양자화한 목표 펄스 및 카운트를 출력한다. 예:
  `motor1:angle 100.00,pulse 1425.72us,count 1150,duration 2.000s`.
  또는 `motor1:pulse 1500.00us,duration 1.000s,delay 0.800s` 형식이며
  여러 모터는 `/`로 구분한다. `duration`은 요청시간과 안전 최소시간 중
  큰 접수 시점 기준 예정 실행시간을 사용한다. 지연 실행 시 시작 위치가
  달라지면 실제 시작 과정에서 안전 최소시간이 더 늘어날 수 있다.
- 기본 포지션은 `SERVO_POSITIONS` const 2차원 배열에 모든 채널의 논리각을
  저장한다. `#GOPOS1T1000` 형식으로 선택하며 포지션 번호는 1부터
  `SERVO_POSITION_COUNT`까지이다. `T`를 생략하면
  `SERVO_POSITION_DEFAULT_TIME_MS`의 2000ms를 사용하고, 지정할 때는
  100~9999ms 범위여야 한다.
- `#GOPOS`는 모든 모터에 같은 요청시간을 적용하지만 각 모터의 안전
  최소시간 자동 연장은 유지하며, 정상 접수 시 일반 각도 명령과 같은 상세
  응답을 출력한다.
- 테스트 모터 선택 명령은 공백 없는 `#TEST<모터번호>` 형식이다.
  `#TEST 5`처럼 입력하면 `TEST_FORMAT` 오류와 올바른 `#TEST5` 예시를
  반환한다.

## 오프셋 저장 요구사항

- 채널별 오프셋은 런타임 배열 `servoOffsetDeg`를 사용하고 하드코딩된
  `DEFAULT_SERVO_OFFSET_DEG`는 NVS 저장값이 없거나 유효하지 않을 때만
  사용한다.
- `#OFFSET#1A-9.45#3A2.7` 형식으로 여러 모터의 오프셋을 한 번에 설정한다.
- 정확히 `#OFFSET`만 입력하면 저장 없이 현재 오프셋을 조회한다. `OK`를
  먼저 출력하고 다음 줄에 모든 모터의 오프셋을 설정 성공 응답과 같은
  형식으로 출력하며, NVS 기록이나 서보 이동을 수행하지 않는다.
- 오프셋 명령은 모든 항목을 검증한 뒤 한 번에 적용하고 저장한다. 일부
  항목만 적용하거나 부분 저장하지 않는다.
- 오프셋 허용 범위는 -180도부터 180도까지이며 단위는 degree이다.
- 오프셋 적용 후 보정각은 채널에 연결된 서보의 실제 물리 가동범위로
  제한한다. 180° 서보는 `0~180°`, 270° 서보는 중앙 구간 양쪽 45° 여유를
  포함한 `-45~225°`를 허용하여 최종 PWM 500~2500us 범위를 사용할 수 있게
  한다.
- ESP32 Preferences의 `servo-bridge` namespace와 `offsets` key에 전체
  float 배열을 blob으로 저장하고 부팅 시 불러온다.
- 오프셋 저장 명령은 서보를 즉시 움직이지 않는다. 이후 각도 명령과 다음
  부팅 초기 위치 계산부터 새 오프셋을 적용한다.
- 저장 성공 시 기존 `OK`를 먼저 출력하고 다음 줄에
  `angle offset => motor1:-4.00,...,motor8:0.00` 형식으로 모든 모터의
  현재 오프셋을 소수점 두 자리까지 출력한다.
- 오프셋 저장 실패나 명령 오류 시에는 전체 오프셋 목록을 출력하지 않는다.
- Raw PWM `P` 명령에는 오프셋을 적용하지 않는다.
- 저장 구조나 `SERVO_COUNT`를 변경할 때는 기존 blob 크기가 맞지 않으면
  기본 오프셋으로 안전하게 복귀하도록 유지한다.

## 하드웨어 요구사항

- 모든 서보의 합산 스톨 전류를 감당할 수 있는 별도 전원으로 서보를
  구동한다.
- ESP32-C3 보드에서 서보 전원 레일에 전원을 공급하지 않는다.
- ESP32-C3, PCA9685 및 서보 전원 공급 장치의 GND를 공통으로 연결한다.
- PCA9685 `VCC`는 ESP32-C3의 3.3V, `V+`는 별도 서보 전원에 연결한다.
- PCA9685 `OE`는 ESP32-C3 GPIO10에 직접 연결하고 PCA9685 `VCC` 3.3V에
  10kΩ으로 풀업한다. ESP32 리셋과 부팅 중에는 HIGH로 출력을 차단하고
  PCA9685 초기화 및 8번 모터의 초기 PWM 등록 후 LOW로 활성화한다.
- 서보 전원을 켜기 전에 PCA9685 로직 전압, I2C 풀업 저항 및 보드 주소를
  확인한다.

## ROS2 Python 브리지 요구사항

- ROS2 Python 패키지는 `ament_python`, `rclpy` 및 `pyserial`을 사용한다.
  Windows와 Ubuntu에서 같은 Python 코드와 ROS2 인터페이스를 유지하고
  포트 이름과 OS별 설정만 파라미터 또는 YAML로 분리한다.
- Windows 시리얼 포트는 `COM3` 같은 이름을 사용하고 Ubuntu는
  `/dev/ttyACM0` 같은 장치 경로를 사용한다. 기본값을 제공하되 실제 포트는
  ROS2 파라미터나 설정 파일로 변경할 수 있어야 한다.
- 자동 포트 검색은 편의 기능으로만 사용한다. 장치가 여러 개이거나 식별이
  불확실하면 임의의 포트를 열지 않고 명시적인 설정을 요구한다.
- 시리얼 설정은 기존 펌웨어와 동일한 `115200 baud, 8N1`을 사용한다.
  명령은 한 줄 단위로 송신하고 ESP32가 출력하는 UTF-8 응답을 줄 단위로
  수신한다.
- 브리지 노드 하나만 시리얼 포트를 소유한다. UI, 시뮬레이터 어댑터 및
  다른 ROS2 노드는 시리얼 포트를 직접 열지 않는다.
- 모든 송신 명령은 단일 큐에서 순서를 보장한다. 동시에 들어온 UI와
  시뮬레이터 명령이 바이트 단위로 섞이지 않도록 한다.
- `BOOT`, `READY`, `OK`, `ERR`, `STATE`, `RESET motor...`, 모터별 상세
  응답, `PWMDBG` 및 테스트 메시지를 구분하여 파싱한다. 알 수 없는 줄도
  버리지 말고 진단 로그에 남긴다.
- 연결이 끊기면 브리지 프로세스를 종료하지 않고 설정 가능한 간격으로
  재연결한다. 재연결 후에는 `READY`를 기다리며 자동 `#RESET`이나 자동
  이동 명령을 보내지 않는다.
- Windows의 포트 점유 또는 장치 제거, Ubuntu의 장치 권한 부족 또는 장치
  제거를 서로 구분 가능한 진단 상태로 보고한다.
- ESP32가 `READY` 상태가 아니거나 이전 명령 응답을 처리할 수 없는
  상태에서는 새 하드웨어 명령을 무제한 누적하지 않는다.
- ROS2 내부 각도 단위는 radian을 사용한다. ESP32 시리얼 프로토콜과
  운영자 UI 표시는 degree와 microsecond를 사용하며 변환은 브리지 경계에서
  명시적으로 수행한다.
- 채널별 ROS 조인트 이름, 표시 이름, 액추에이터 종류, 모터 번호, 논리각
  제한 및 UI 노출 여부를 YAML 설정으로 관리한다. 배열 순서에만 의존하지
  말고 모터 번호와 이름의 중복을 시작 시 검증한다.
- ROS2 명령 입력은 `/servo_bridge/joint_trajectory`의
  `trajectory_msgs/msg/JointTrajectory`를 기본 인터페이스로 사용한다.
  첫 구현은 첫 번째 trajectory point를 실행하며 `time_from_start`를
  ESP32 명령의 `T` 값으로 변환한다.
- trajectory에서 조인트 이름 순서는 자유롭게 허용하되 알 수 없는 이름,
  중복 이름, 값 개수 불일치, NaN/Inf 및 설정 범위를 벗어난 값은 ESP32로
  보내기 전에 전체 명령을 거부한다.
- ESP32의 100~9999ms 수행시간 제한을 지킨다. ROS2에서 100ms보다 짧은
  요청은 100ms로 제한하고 9999ms를 초과하는 요청은 오류로 거부한다.
- ESP32 `STATE`를 파싱하여 `/servo_bridge/joint_states`에
  `sensor_msgs/msg/JointState`로 발행한다. position은 radian으로
  변환하고 실제 센서 측정값으로 오해하지 않도록 진단 상태에 open-loop임을
  포함한다.
- 연결 상태, `READY` 여부, 마지막 송수신 시각, ESP32 오류 및 재연결 상태는
  `/servo_bridge/diagnostics`에
  `diagnostic_msgs/msg/DiagnosticArray`로 발행한다.
- `/servo_bridge/reset`은 `std_srvs/srv/Trigger` 서비스로 제공한다.
  서비스 성공은 물리 이동 완료가 아니라 ESP32가 `#RESET`을 정상 접수한
  것을 뜻한다.
- 시뮬레이터의 고주파 명령은 설정 가능한 최대 송신률로 제한한다. 아직
  송신하지 않은 연속 trajectory 명령은 최신 값을 우선하여 병합하되
  `#RESET`, 오프셋 저장 같은 관리 명령은 버리거나 재정렬하지 않는다.

## 직접 제어 UI 요구사항

- UI는 PySide6 데스크톱 애플리케이션으로 구현하고 ROS2 브리지와 별도
  프로세스로 실행한다. UI가 종료되거나 재시작되어도 브리지와 시리얼
  연결은 유지되어야 한다.
- UI는 ROS2 토픽과 서비스만 사용하며 시리얼 포트에 직접 접근하지 않는다.
- 각 채널의 모터 번호, 설정 이름, 액추에이터 종류, 현재 계산 각도,
  현재 계산 펄스, 목표 각도, 목표 펄스 및 이동 상태를 한 화면에서 확인할
  수 있어야 한다.
- 현재값은 ESP32 `STATE`에서 받은 값으로 표시한다. 목표값은 로컬 계산값이
  아니라 ESP32가 정상 접수 후 반환한 모터별 상세 응답을 우선 사용하여
  PCA9685 카운터 양자화가 적용된 목표 펄스를 표시한다.
- 채널별 각도 슬라이더와 숫자 입력, 공통 또는 채널별 수행시간, 선택 채널
  전송, 전체 전송 및 Reset 기능을 제공한다.
- 각 입력은 YAML에 정의된 논리각 제한을 적용한다. 그리퍼와 보조
  액추에이터도 관절 전용 표현을 강제하지 않고 설정된 이름과 종류로
  표시한다.
- 연결 포트, 연결 여부, ESP32 `READY` 여부, 마지막 명령, 마지막 `OK` 또는
  `ERR`, 마지막 수신 시각 및 open-loop 경고를 항상 확인할 수 있어야 한다.
- UI의 송신 버튼은 브리지의 명령 접수 결과를 표시하며 `OK`를 물리 이동
  완료로 표현하지 않는다.
- Windows와 Ubuntu에서 같은 화면 구성과 기능을 유지하고 OS별 경로 또는
  포트 이름을 UI 코드에 하드코딩하지 않는다.

## ROS2 브리지 개발 로드맵

1. ROS2 Jazzy `ament_python` 패키지, 공통 YAML 설정, Windows 및 Ubuntu
   실행 문서와 launch 구성을 만든다.
2. 순수 Python 시리얼 코어와 응답 파서를 먼저 구현하고 가상 시리얼
   환경에서 부분 수신, UTF-8 오류, 재연결 및 명령 큐 동작을 검증한다.
3. 시리얼 코어를 ROS2 노드에 연결하여 `JointTrajectory`, `JointState`,
   diagnostics 및 Reset 서비스를 구현한다.
4. PySide6 UI를 별도 ROS2 프로세스로 구현하고 현재값, 목표값, 연결 상태와
   직접 제어 기능을 제공한다.
5. Windows 11의 실제 COM 포트와 ESP32에서 연결, Reset, 단일 및 다중 채널
   이동, 장치 제거와 재연결을 우선 검증한다.
6. Ubuntu 24.04의 `/dev/ttyACM*` 환경에서 같은 YAML과 ROS2 인터페이스로
   동일 동작을 검증한다.
7. Isaac Sim과 MuJoCo 어댑터가 동일한 `JointTrajectory` 인터페이스를
   사용하도록 연결하고 고주파 명령 제한을 검증한다.
8. 단일 point 제어가 안정화된 뒤 다중 trajectory point 실행과
   `ros2_control` 연동을 후속 단계로 추가한다.

## ROS2 브리지 검증 기준

- radian/degree 변환, 채널 순서 변경, 제한 범위, 중복 이름 및 7·8번
  비관절 액추에이터 설정을 단위 테스트한다.
- `JointTrajectory` 입력이 올바른 다중 채널 ESP32 명령과 `T` 값으로
  변환되는지 검증한다.
- `STATE`와 모터별 상세 응답이 현재 각도/us 및 목표 각도/us로 정확히
  분리되는지 검증한다.
- UI 종료 후에도 브리지 노드와 시리얼 연결이 유지되는지 검증한다.
- Windows 10/11과 Ubuntu 24.04에서 동일한 설정 구조와 ROS2 토픽/서비스
  계약을 유지한다.
- 실제 하드웨어 테스트에서는 센서 측정값이 아닌 계산 상태만 제공된다는
  점과 `OK`가 이동 완료가 아니라 명령 접수임을 확인한다.
