# 서보 브리지 개발 지침

## 프로젝트 목적

- 이 저장소는 ESP32-C3 펌웨어와 ROS2 Python 브리지를 함께 개발하여 실제 서보 장치, 직접 제어 UI, 로봇 시뮬레이터를 같은 제어 경로로 연결한다.
- 전체 데이터 흐름은 `Isaac Sim 또는 MuJoCo -> ROS2 -> Python 브리지 -> USB Serial -> ESP32-C3 -> Servo GPIO PWM`이다.
- ROS2 기준 배포판은 Jazzy이며 Windows 10/11과 Ubuntu 24.04를 지원한다. 현재 주 개발 및 실제 장치 검증 환경은 Windows이다.
- ESP32 시리얼 프로토콜을 실제 하드웨어 제어의 단일 기준으로 유지한다. ROS2 노드와 UI는 서보를 직접 제어하지 않는다.
- 각 모터는 관절뿐 아니라 그리퍼나 보조 액추에이터일 수 있다. 축 또는 관절이라는 가정은 설정 파일의 `type`으로만 표현한다.
- ESP32의 `STATE`와 ROS2 `JointState`는 센서 측정값이 아니라 펌웨어가 계산한 open-loop 상태임을 명확히 표시한다.

## ESP32 직접 서보 출력 요구사항

- PCA9685는 사용하지 않는다. I2C, enable 핀, PCA9685 카운터 변환 코드를 추가하지 않는다.
- 기존 시리얼 명령 프로토콜과 모터 번호 체계를 유지한다.
- 기본 대상은 ESP32-C3 6채널이다. 기본 핀은 motor1~6 순서로 GPIO `7, 8, 9, 10, 20, 21`이다.
- 다른 ESP32 보드에서 채널을 늘릴 때는 `SERVO_COUNT`, `SERVO_PINS`, 모든 축별 설정 배열, ROS2 `servos.yaml`을 함께 늘린다.
- 출력은 ServoEasing과 ESP32Servo 기반으로 50 Hz PWM을 사용한다.
- 목표는 50 Hz LEDC 16-bit 해상도이다. 해당 보드/라이브러리 조합에서 16-bit가 지원되지 않으면 낮은 해상도로 자동 대체하지 말고 실패를 보고한다.
- 펄스 폭은 microsecond 단위로 관리한다. PCA9685 count 또는 channel 0 예약 규칙을 다시 도입하지 않는다.
- 각도 명령은 축별 가동 범위, 방향, 오프셋, 초기 위치 및 각도 제한 설정을 유지한다.
- Raw PWM `P` 명령은 500~2500us 범위의 물리 펄스를 직접 지정하며 오프셋을 적용하지 않는다.
- ServoEasing은 non-blocking 방식으로 사용한다. 시리얼 수신과 지연 명령 처리를 막는 blocking easing 호출을 사용하지 않는다.
- easing 타입은 기본적으로 `EASE_SINE_IN_OUT`을 사용한다.
- 채널별 독립 수행시간, 지연 실행, 안전 최소 이동시간 계산을 유지한다.
- 데드밴드와 마지막 보정은 microsecond 단위로 판단한다.
- 일반 루프의 모션 갱신 시각은 시리얼 명령 처리가 끝난 뒤 읽는다. 새 명령의 모션 시작 시각보다 오래된 `millis()` 값으로 갱신하여 unsigned 시간 계산이 언더플로되지 않도록 유지한다.

## 부팅과 리셋

- 부팅과 `#RESET`은 `SERVO_RESET_ORDER`에 정의한 순서로 초기화한다. 기본 6채널 순서는 2, 3, 4, 5, 6, 1이다.
- 각 모터의 이동 시작 간격은 `SERVO_RESET_STAGGER_MS` 값이며 현재 기본값은 800ms이다.
- 부팅 시 마지막 1번 모터의 초기 PWM 인가가 시작된 뒤 `READY`를 출력하며, 그전에는 시리얼 명령을 처리하지 않는다.
- `#RESET`은 진행 중인 모션과 모든 지연 명령을 현재 계산 위치에서 취소한 뒤 순차 리셋을 예약하고 `OK`를 즉시 반환한다.
- `#RESET`으로 예약된 각 모터가 실제로 시작될 때 `RESET motor2 angle:90.00` 형식의 메시지를 `SERVO_RESET_ORDER` 순서로 출력한다. 부팅 초기화와 일반 지연 명령에서는 이 메시지를 출력하지 않는다.
- `PWM_INITIALIZATION_DEBUG`가 활성화되면 부팅 초기화와 각 `#RESET`에서 실제 서보 출력 attach/write/ease를 `PWMDBG` 형식으로 출력한다. 일반 제어 명령은 출력하지 않는다.

## 시리얼 프로토콜

- 오류 응답은 `ERR,<영문 코드>,<한국어 설명>,INPUT=<입력>` 형식을 유지하고 시리얼 클라이언트는 UTF-8로 디코딩한다.
- 일반 각도 및 Raw PWM 명령이 정상 등록되면 `OK` 다음 줄에 선택된 모터를 번호순으로 출력한다.
- 각도 항목은 입력 각도, 출력 GPIO 핀, 목표 펄스를 출력한다. 예: `motor1:angle 100.00,pin 7,pulse 1426.00us,duration 2.000s`.
- Raw PWM 항목은 `motor1:pulse 1500.00us,pin 7,duration 1.000s,delay 0.800s` 형식을 사용한다.
- 여러 모터는 `/`로 구분한다.
- `duration`은 요청시간과 안전 최소시간 중 큰 접수 시점 기준 예정 실행시간을 사용한다. 지연 실행 시 시작 위치가 달라지면 실제 시작 과정에서 안전 최소시간이 더 늘어날 수 있다.
- 기본 포지션은 `SERVO_POSITIONS` const 2차원 배열에 모든 모터의 논리각을 저장한다. `#GOPOS1T1000` 형식으로 선택하며 포지션 번호는 1부터 `SERVO_POSITION_COUNT`까지이다.
- `#GOPOS`에서 `T`를 생략하면 `SERVO_POSITION_DEFAULT_TIME_MS`의 2000ms를 사용하고, 지정할 때는 100~9999ms 범위여야 한다.
- 테스트 모터 선택 명령은 공백 없는 `#TEST<모터번호>` 형식이다. `#TEST 5`처럼 입력하면 `TEST_FORMAT` 오류와 올바른 `#TEST5` 예시를 반환한다.
- `#PINMAP`은 저장이나 서보 이동 없이 `OK` 다음 줄에 `PINMAP,motor1:gpio7,...` 형식으로 현재 모터 번호와 GPIO 핀 매핑을 출력한다.

## 오프셋 저장 요구사항

- 축별 오프셋은 런타임 배열 `servoOffsetDeg`를 사용하고 하드코딩된 `DEFAULT_SERVO_OFFSET_DEG`는 NVS 저장값이 없거나 유효하지 않을 때만 사용한다.
- `#OFFSET#1A-9.45#3A2.7` 형식으로 여러 모터의 오프셋을 한 번에 설정한다.
- 정확히 `#OFFSET`만 입력하면 저장 없이 현재 오프셋을 조회한다. `OK`를 먼저 출력하고 다음 줄에 모든 모터의 오프셋을 설정 성공 응답과 같은 형식으로 출력하며, NVS 기록이나 서보 이동을 수행하지 않는다.
- 정확히 `#OFFSET#RESET`을 입력하면 Preferences의 `offsets` key를 삭제하고 런타임 오프셋을 기본값으로 되돌린다. 서보를 즉시 움직이지 않으며 성공 시 `OK` 다음 줄에 현재 기본 오프셋을 출력한다.
- 오프셋 명령은 모든 항목을 검증한 뒤 한 번에 적용하고 저장한다. 일부 항목만 적용하거나 부분 저장하지 않는다.
- 오프셋 허용 범위는 -180도부터 180도까지이며 단위는 degree이다.
- 오프셋 적용 후 보정각은 축의 실제 물리 가동범위로 제한한다. 180도 축은 `0~180도`, 270도 축은 중앙 구간 양쪽 45도 여유를 포함한 `-45~225도`를 허용하여 최종 PWM 500~2500us 범위를 사용할 수 있게 한다.
- ESP32 Preferences의 `servo-bridge` namespace와 `offsets` key에 float 배열을 blob으로 저장하고 부팅 시 불러온다.
- 부팅 시 저장된 blob 크기가 `sizeof(float)`의 배수이면 저장된 float 개수만큼 읽되 현재 사용하는 `SERVO_COUNT` 앞쪽 축에만 적용한다. 저장된 값이 현재 축보다 적으면 나머지는 기본 오프셋을 유지하고, 저장된 값이 현재 축보다 많으면 뒤쪽 값은 무시한다.
- 저장된 blob 크기가 깨졌거나 적용 대상 값 중 하나라도 NaN/Inf 또는 허용 범위 밖이면 전체 저장값을 무시하고 기본 오프셋으로 안전하게 복귀한다.
- 오프셋 저장 명령은 서보를 즉시 움직이지 않는다. 이후 각도 명령과 다음 부팅 초기 위치 계산부터 새 오프셋을 적용한다.
- 저장 성공 시 기존 `OK`를 먼저 출력하고 다음 줄에 `angle offset => motor1:-4.00,...,motor6:0.00` 형식으로 모든 모터의 현재 오프셋을 소수점 두 자리까지 출력한다.
- 오프셋 저장 실패나 명령 오류 시에는 전체 오프셋 목록을 출력하지 않는다.
- 저장 구조나 `SERVO_COUNT`를 변경해도 기존 blob의 앞쪽 축 값은 가능한 한 유지되도록 한다.

## 하드웨어 요구사항

- 모든 서보의 합산 스톨 전류를 감당할 수 있는 별도 전원으로 서보를 구동한다.
- ESP32-C3 보드에서 서보 전원 레일에 전원을 공급하지 않는다.
- ESP32-C3와 서보 전원 공급 장치의 GND를 공통으로 연결한다.
- 서보 신호선은 `SERVO_PINS` 배열의 GPIO 순서대로 연결한다.
- ESP32 리셋 순간에는 GPIO가 high-impedance나 부트 스트랩 상태가 될 수 있으므로 직접 GPIO 방식만으로 리셋 중 무신호를 완전히 보장하지 않는다. 필요하면 서보 전원 스위칭, 외부 풀다운, 버퍼 enable 회로를 추가한다.
- GPIO 8, 9, 20, 21은 보드에 따라 내장 LED, 기본 I2C, UART 표기와 겹칠 수 있다. 이 프로젝트에서는 PCA9685와 UART 시리얼을 쓰지 않으며 USB CDC 시리얼을 사용한다.

## ROS2 Python 브리지 요구사항

- ROS2 Python 패키지는 `ament_python`, `rclpy`, `pyserial`을 사용한다.
- Windows 시리얼 포트는 `COM3` 같은 이름을 사용하고 Ubuntu는 `/dev/ttyACM0` 같은 경로를 사용한다. 실제 포트는 ROS2 파라미터나 YAML로 바꿀 수 있어야 한다.
- 시리얼 설정은 `115200 baud, 8N1`을 사용한다.
- 브리지 노드 하나만 시리얼 포트를 소유한다. UI, 시뮬레이터 어댑터 및 다른 ROS2 노드는 시리얼 포트를 직접 열지 않는다.
- `BOOT`, `READY`, `OK`, `ERR`, `STATE`, `RESET motor...`, 모터별 상세 응답, `PWMDBG` 및 테스트 메시지를 구분하여 파싱한다. 알 수 없는 줄도 버리지 말고 진단 로그에 남긴다.
- 연결이 끊기면 브리지 프로세스를 종료하지 않고 설정 가능한 간격으로 재연결한다. 재연결 후에는 `READY`를 기다리며 자동 `#RESET`이나 자동 이동 명령을 보내지 않는다.
- ROS2 내부 각도 단위는 radian을 사용한다. ESP32 시리얼 프로토콜과 UI 표시에는 degree와 microsecond를 사용하며 변환은 브리지 경계에서 명시적으로 수행한다.
- 채널별 ROS 조인트 이름, 표시 이름, 액추에이터 타입, 모터 번호, 논리각 제한 및 UI 노출 여부를 YAML 설정으로 관리한다.
- ROS2 명령 입력은 `/servo_bridge/joint_trajectory`의 `trajectory_msgs/msg/JointTrajectory`를 기본 인터페이스로 사용한다.
- ESP32 `STATE`를 파싱하여 `/servo_bridge/joint_states`의 `sensor_msgs/msg/JointState`로 발행한다. position은 radian으로 변환하고 실제 센서 측정값으로 오해하지 않도록 진단 상태에 open-loop임을 포함한다.
- 연결 상태, `READY` 여부, 마지막 송수신 시각, ESP32 오류 및 재연결 상태를 `/servo_bridge/diagnostics`의 `diagnostic_msgs/msg/DiagnosticArray`로 발행한다.
- `/servo_bridge/reset`은 `std_srvs/srv/Trigger` 서비스로 제공한다. 서비스 성공은 물리 이동 완료가 아니라 ESP32가 `#RESET`을 정상 접수한 것을 뜻한다.

## 직접 제어 UI 요구사항

- UI는 PySide6 데스크톱 애플리케이션으로 구현하고 ROS2 브리지와 별도 프로세스로 실행한다.
- UI는 ROS2 토픽과 서비스만 사용하며 시리얼 포트에 직접 접근하지 않는다.
- 각 채널의 모터 번호, 설정 이름, 액추에이터 타입, 현재 계산 각도, 현재 계산 펄스, 목표 각도, 목표 펄스 및 이동 상태를 한 화면에서 확인할 수 있어야 한다.
- 현재값은 ESP32 `STATE`에서 받은 값으로 표시한다. 목표값은 ESP32가 정상 접수 후 반환한 모터별 상세 응답을 우선 사용한다.
- 채널별 각도 슬라이더와 숫자 입력, 공통 또는 채널별 수행시간, 선택 채널 전송, 전체 전송 및 Reset 기능을 제공한다.
- 그리퍼나 보조 액추에이터도 관절용 표현을 강제하지 않고 설정의 이름과 타입으로 표시한다.

## ROS2 브리지 개발 로드맵

1. ROS2 Jazzy `ament_python` 패키지, 공통 YAML 설정, Windows 및 Ubuntu 실행 문서와 launch 구성을 유지한다.
2. Python 시리얼 코어와 응답 파서를 실제 ESP32 프로토콜 변경에 맞춰 유지한다.
3. `JointTrajectory`, `JointState`, diagnostics, Reset 서비스를 안정화한다.
4. PySide6 UI에서 현재값, 목표값, 연결 상태와 직접 제어 기능을 유지한다.
5. Windows 11 실제 COM 포트와 ESP32에서 연결, Reset, 단일 및 다중 채널 이동, 장치 제거와 재연결을 우선 검증한다.
6. Ubuntu 24.04 `/dev/ttyACM*` 환경에서 같은 YAML과 ROS2 인터페이스로 동일 동작을 검증한다.
7. Isaac Sim과 MuJoCo 어댑터가 동일한 `JointTrajectory` 인터페이스를 사용하도록 연결한다.

## 검증 기준

- Arduino CLI로 `esp32:esp32:nologo_esp32c3_super_mini` 대상 컴파일이 성공해야 한다.
- 50 Hz 16-bit LEDC 설정이 낮은 해상도로 조용히 강등되지 않아야 한다.
- radian/degree 변환, 채널 순서 변경, 제한 범위, 중복 이름, 비관절 액추에이터 설정을 단위 테스트한다.
- `JointTrajectory` 입력이 올바른 다중 모터 ESP32 명령과 `T` 값으로 변환되는지 검증한다.
- `STATE`와 모터별 상세 응답이 현재 각도/us 및 목표 각도/us로 정확히 분리되는지 검증한다.
- 실제 하드웨어 테스트에서는 계산 상태만 제공한다는 점과 `OK`가 이동 완료가 아니라 명령 접수임을 확인한다.
